## 2. 감사 발견

### A. 인프라 / CI (P0 — 지금 실제로 고장 나 있는 것)

- **A1. 스케줄 CI fit-smoke 상시 실패.** `.github/workflows/build.yml`의 nightly `fit-pipeline-smoke` 잡이 **`ModuleNotFoundError: numpy`로 매일 실패** 중(런 29604407966 로그 확증, 최소 2일 연속). 워크플로에 Python 의존성 설치 스텝이 없다 — 로컬에서 겪은 numpy 환경 문제와 같은 계급이며, 로컬은 `sys.executable`로 견고화했지만 CI는 numpy 자체가 없다. **수정:** `pip3 install numpy scipy` 스텝 1개(scipy는 선택 의존 — 없으면 힐클라임 폴백이지만 적합 품질을 위해 포함).
- **A2. CI가 전체 테스트의 ~80%만 게이트.** 푸시 잡이 `VALVRA_BUILD_PLUGIN=OFF`로 구성되어 **플러그인 테스트 39개(스트레스 스위트 포함)와 플러그인 빌드 자체가 CI 미커버**. 2m22s의 "빠른 그린"은 축소 범위의 결과. 이번 사이클에서 스트레스 스위트가 잡아낸 결함들(피드스루 발진, 플럭스 불안정, carry 스텝)을 생각하면 이 공백은 크다. **수정:** JUCE 페치 캐시 포함 플러그인 잡 추가(또는 기존 잡 PLUGIN=ON 전환 + actions/cache).
- **A3. 인-소스 빌드 오염.** 루트/`src`/`tests`에 CMakeCache·CMakeFiles·바이너리 등 40여 미추적 파일(과거 인-소스 cmake 실행의 잔재). `.gitignore`는 `build/`만 무시. **수정:** 패턴 추가 + `git clean` 1회 + 인-소스 구성 방지(`CMAKE_DISABLE_IN_SOURCE_BUILD`).
- **A4. Python 환경 취약 잔재.** 파이프라인 내부는 `sys.executable`로 고정했지만, 진입점 자체와 `practical_readiness_report.py` 직접 실행은 여전히 PATH의 `python3`에 의존(Homebrew 3.14가 시스템 3.9를 가리는 환경에서 재발). **수정:** 스크립트 셔뱅/README 명시 + CI에서는 A1로 해소.

### B. 검증 체계의 구조적 한계 (P1 — 게이트가 말해주지 않는 것)

- **B1. Fit 게이트의 자기참조성.** readiness의 Objective RMSE가 **5모드 전부 0.00000** — "external" 캡처가 `--bootstrap-external-from-internal`로 **엔진 자신에게서 생성**되기 때문. 즉 현행 Fit Gate는 *하드웨어 근접도*가 아니라 **적합 파이프라인의 자기일관성**(엔진→데이터셋→재적합→재현)을 검증한다. 원 기획(docs/22 "실물 측정 없이 구현 가능 수준")과 정합하는 의도된 구조지만, 게이트 명칭이 과신을 유발한다. **수정:** (i) 리포트에 "self-referential" 라벨 명시, (ii) 장기적으로 실물 캡처 1대 확보(docs/18)로 진짜 외부 기준 승격 — §3 S3.
- **B2. 시드 널 미재측정.** docs/33 §4는 시드 널(v72 −18.3 … marshall −24.3 dB, 목표 −40…−10)을 추적했지만, W1–W8 동안 어떤 게이트도 이를 재측정하지 않았다(readiness 리포트에 부재 확인). MC 스트림이 W5에서 시프트되고 W3/W4가 시드 의존 물리를 늘렸으므로 현재 값은 미지. **수정:** readiness 리포트에 시드 널 섹션 복원 + 목표 범위 게이트화.
- **B3. 골든 렌더 회귀 부재.** 각 웨이브는 소리를 *의도적으로* 바꿨고 refit이 이를 흡수했다 — 그 결과 **비의도 드리프트를 잡는 장치가 없다**(refit은 무엇이든 다시 적합해버린다). **수정:** 프리셋×드라이브 2점의 스펙트럼 지문(H1–H5, RMS, 시드 고정)을 아티팩트로 고정하고 ctest 가드(±0.5 dB)로 잠금 — "의도 변경 시 지문 갱신 + 커밋 메시지에 사유"를 규약화. 이것이 refit과 상보적인 *변화 탐지* 게이트가 된다.
- **B4. 벤치 절차 무통제.** 이번 감사에서 `mediaanalysisd`(사진 인덱싱)가 90% CPU를 점유한 상태의 벤치가 HiFi 492%/CV 168% 같은 5–7× 오염 수치를 산출했고, 재실행 간 편차가 2×를 넘었다. 회귀 판정에 쓰기엔 위험. **수정:** 벤치 러너 스크립트화 — 유휴 검사(CPU>20% 프로세스 감지 시 경고/대기), 3회 실행 최소값 채택, 결과를 docs/27 이력에 자동 append.

### C. 물리 잔존 이음새 (P2 — 다음 물리 사이클 후보)

- **C1.** `mainsFrequencyHz` 50/60 스위치가 엔진에만 존재, **UI/CLI 파라미터 미노출**(항상 60 Hz — EU 보이싱 불가). 노출 시 1파라미터.

- **C4.** UL 탭이 법칙 평가만 바꿈 — **스크린 전류의 분배**(테일/PSU 도달, 스크린 새그)는 미모델. UL을 실제 프리셋에 쓸 때 함께 승격할 것. 현재 어떤 프리셋도 미사용(기능만 존재).

- **C6.** 신설 물리 계수들이 공학 추정치: `divK/divV`(Ig 분할), `kExcess`(Bertotti), 리액티브 부하 파라미터, PP 그리드 RC. 문헌 재보정 여지(Rutt 1984 division 곡선, Bertotti 손실 분해 데이터).
- **C7.** `CathodeBounce::primeTo`가 soakage 분리(Vs≠Vk)를 소실 — carry 시 DA "블룸" 상태가 평탄화됨(미세).
- **C8.** 리액티브 부하 SE 한정(설계 의도) — Console PP의 리액티브(라인+트랜스 리액턴스)는 미결정 항목으로 유지.
- **C9.** `AnalogInteraction` 휴리스틱 레이어 잔존 — 정책상 종속층이나, **NFB 실루프 도입 후 `feedbackAmount` 휴리스틱과 역할 중복**. 실루프 기반으로 통합·격하 검토.

6. **B1 라벨링**: readiness 리포트 Fit Score에 "self-referential (engine-bootstrapped)" 명시.

### S2 — 물리 마감(잔여 이음새, ~1주)

**D-A/D-B (CV 엔진 결함, S1 게이트가 발견 — 최우선)** → C1 mains UI/CLI 파라미터 → C2 펜토드 Rg → C3 α(t) 추종 → C7 soakage 보존 carry → C10 CV 바이어스 직결 → C9 휴리스틱 feedback 레이어를 NFB 실루프로 통합·격하. 각각 가드 테스트 + (음색 변경 건은) 골든 지문 갱신 + refit.

- **D-A 쌍안정 스크린 노드**: CV 2단(6AS6, 스크린 급전 저항 큼)에서 `Rs·dIg2/dVg2 > 1` 구간이 존재 → setup의 rest 이분법이 **불안정 근**에 안착할 수 있다(예: seed 0x11112222 rest Ip 0.082 mA vs 런타임 수렴 0.032 mA — 약 2.4 V의 상시 오프셋). W8 볼트-네이티브가 rest를 정적 앵커로 쓰면서 표면화됐다. 수정 방향: rest 산출을 런타임과 동일한 스크린 RC 평형(고정점 반복)으로 통일.
- **D-B OPT H-레일 리밋사이클**: D-A의 상시 DC가 OPT를 심포화로 몰면(seed 777: H≈52a, M≥Ms) JA 스텝의 ±64a 클램프 경계에서 나이퀴스트 진동이 상주(|y|=59.7 상수 진폭). 수정 방향: 심포화 영역에서 클램프가 아니라 dM/dH→0의 물리로 자연 포화되도록 상한 처리 재설계.
- **실패한 픽스 3건의 교훈(재시도 금지 목록)**: ① rest 재앵커 settle 루프 — 런타임 완화로 rest만 옮기면 전 체인 캘리브레이션(패드·NFB β)이 깨진다. ② in-솔버 h 클램프 — 심포화의 거대 자화전류 드롭은 **물리적 리미팅 그 자체**라 절단하면 출력이 폭주한다(CV rms 0.0009→15). ③ 플럭스 anti-windup — 정상 포화도 demand>achievable 오버슈트가 필수라 오판한다. 세 건 모두 리버트, 엔진은 main과 비트 동일.

**시드 널 방법론의 진화** — 원안(원시 널 −45…−6 dB 단일 창)은 두 번 반증됐다:

1. **노이즈 스트림 오염**: 샷/험 발생기는 시드별 독립 스트림이라(설계상 무상관) 조용한 체인에서는 캐릭터와 무관하게 널을 0 dB로 끌어올린다 → 측정 시 노이즈 비활성.
2. **평탄 게인 산포와 캐릭터의 혼합**: HiFi 원시 널 −3.5 dB의 정체는 위상(2.7°)도 파형도 아닌 **유닛 간 게인 1.67×(+4.5 dB)** 였다(300B G-µ 산포의 정당한 개별성). → 2단 판정으로 분리: **원시 널 > −45 dB**(개별성 하한) ∧ **최소자승 레벨매칭 후 널 < −6 dB**(캐릭터 상한) ∧ **|게인 편차| < 6 dB**(서비스 허용 창). 현행 4프리셋: 원시 −3.5…−15.7, 매칭 후 −23.8…−38.2, 게인 ≤4.4 dB — 전부 창 안.

**CV는 게이트에서 잠정 제외** — §S2 D-A/D-B 결함이 몬테카를로 코너에서 깨진 유닛을 만들 수 있고, 해당 드라이브에서 베어 체인 출력이 노이즈 지배(−57 dBFS)라 널 자체가 무의미. D-A/D-B 수정 사이클에서 레벨 설계와 함께 재편입한다.
